## Gap Analysis Report Schema

The gap analysis report schema defines the structure of `gap_analysis_report.json` files generated by the analyzer.

### Top-Level Structure

```json
{
  "analysis_date": "2024-03-15T14:30:00Z",
  "input_file": "isms_policy.pdf",
  "input_file_hash": "a1b2c3d4e5f6...",
  "model_name": "qwen2.5-3b-instruct",
  "model_version": "1.0.0",
  "embedding_model": "all-MiniLM-L6-v2",
  "gaps": [...],
  "covered_subcategories": [...],
  "metadata": {...}
}
```

### Field Descriptions

#### Root Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `analysis_date` | string (ISO 8601) | Yes | Timestamp when analysis was executed |
| `input_file` | string | Yes | Path to the analyzed policy document |
| `input_file_hash` | string | Yes | SHA-256 hash of input file for traceability |
| `model_name` | string | Yes | LLM model name used for analysis |
| `model_version` | string | Yes | LLM model version/quantization info |
| `embedding_model` | string | Yes | Embedding model name (e.g., "all-MiniLM-L6-v2") |
| `gaps` | array | Yes | Array of gap detail objects |
| `covered_subcategories` | array | Yes | Array of CSF subcategory IDs that are covered |
| `metadata` | object | Yes | Additional analysis metadata |

#### Gap Detail Object

Each object in the `gaps` array has the following structure:

```json
{
  "subcategory_id": "GV.SC-01",
  "description": "Cyber supply chain risk management processes are identified...",
  "status": "missing",
  "evidence_quote": "",
  "gap_explanation": "The policy does not address supply chain cybersecurity risks...",
  "severity": "critical",
  "suggested_fix": "Add a section on 'Supply Chain Risk Management'..."
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `subcategory_id` | string | Yes | CSF subcategory identifier (pattern: `^[A-Z]{2}\.[A-Z]{2}-[0-9]{2}$`) |
| `description` | string | Yes | Full NIST CSF subcategory description text |
| `status` | string | Yes | Gap status: `"partially_covered"` or `"missing"` |
| `evidence_quote` | string | Yes | Relevant text from policy (empty string if missing) |
| `gap_explanation` | string | Yes | Explanation of what is missing or inadequate |
| `severity` | string | Yes | Gap severity: `"critical"`, `"high"`, `"medium"`, or `"low"` |
| `suggested_fix` | string | Yes | Suggested policy language to address the gap |

#### Metadata Object

```json
{
  "prompt_version": "1.0",
  "config_hash": "a1b2c3d4e5f6",
  "retrieval_params": {
    "chunk_size": 512,
    "overlap": 50,
    "top_k": 5
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `prompt_version` | string | Yes | Version of prompt templates used |
| `config_hash` | string | Yes | Hash of configuration parameters for reproducibility |
| `retrieval_params` | object | Yes | Retrieval configuration parameters |

## Implementation Roadmap Schema

The implementation roadmap schema defines the structure of `implementation_roadmap.json` files.

### Top-Level Structure

```json
{
  "roadmap_date": "2024-03-15T14:30:00Z",
  "policy_analyzed": "isms_policy.pdf",
  "immediate_actions": [...],
  "near_term_actions": [...],
  "medium_term_actions": [...],
  "metadata": {...}
}
```

### Field Descriptions

#### Root Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `roadmap_date` | string (ISO 8601) | Yes | Timestamp when roadmap was generated |
| `policy_analyzed` | string | Yes | Path to the analyzed policy document |
| `immediate_actions` | array | Yes | Actions for 0-3 months (Critical/High severity) |
| `near_term_actions` | array | Yes | Actions for 3-6 months (Medium severity) |
| `medium_term_actions` | array | Yes | Actions for 6-12 months (Low severity) |
| `metadata` | object | Yes | Additional roadmap metadata |

#### Action Item Object

Each action item in the timeframe arrays has the following structure:

```json
{
  "action_id": "ACT-001",
  "timeframe": "immediate",
  "severity": "critical",
  "effort": "high",
  "csf_subcategory": "GV.SC-01",
  "policy_section": "New Section 4",
  "description": "Establish Supply Chain Risk Management Program",
  "technical_steps": [
    "Implement vendor risk assessment platform",
    "Develop security questionnaire templates"
  ],
  "administrative_steps": [
    "Define vendor security requirements",
    "Create vendor assessment procedures"
  ],
  "physical_steps": []
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action_id` | string | Yes | Unique action identifier (e.g., "ACT-001") |
| `timeframe` | string | Yes | Implementation timeframe: `"immediate"`, `"near_term"`, or `"medium_term"` |
| `severity` | string | Yes | Gap severity: `"critical"`, `"high"`, `"medium"`, or `"low"` |
| `effort` | string | Yes | Implementation effort: `"low"`, `"medium"`, or `"high"` |
| `csf_subcategory` | string | Yes | CSF subcategory ID being addressed (pattern: `^[A-Z]{2}\.[A-Z]{2}-[0-9]{2}$`) |
| `policy_section` | string | Yes | Policy section requiring changes |
| `description` | string | Yes | Action description |
| `technical_steps` | array of strings | Yes | List of technical implementation steps |
| `administrative_steps` | array of strings | Yes | List of administrative/policy steps |
| `physical_steps` | array of strings | Yes | List of physical security steps (may be empty) |

### Complete Schema Definition

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "roadmap_date": {

    "policy_analyzed": {

      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "action_id": {

          "timeframe": {
            "type": "string",
            "enum": ["immediate", "near_term", "medium_term"]
          },

            "type": "string",

          },
          "effort": {
            "type": "string",

            "type": "string",
            "pattern": "^[A-Z]{2}\\.[A-Z]{2}-[0-9]{2}$"
          },
          "policy_section": {
            "type": "string"
          },
          "description": {
            "type": "string"
          },
          "technical_steps": {
            "type": "array",
            "items": {"type": "string"}
          },
          "administrative_steps": {
            "type": "array",
            "items": {"type": "string"}
```

## Schema Validation

### Python Validation

The system includes built-in validation functions in `models/schemas.py`:

```python
from models.schemas import (
    validate_gap_analysis_report,
    validate_implementation_roadmap,
    is_valid_gap_analysis_report,
    is_valid_implementation_roadmap,
    get_schema_errors
)

# Validate gap analysis report
try:
    validate_gap_analysis_report(report_data)
    print("Report is valid!")
except ValidationError as e:
    print(f"Validation error: {e.message}")

# Check if roadmap is valid
if is_valid_implementation_roadmap(roadmap_data):
    print("Roadmap is valid!")
else:
    errors = get_schema_errors(roadmap_data, IMPLEMENTATION_ROADMAP_SCHEMA)
    print(f"Validation errors: {errors}")
```

### Command-Line Validation

Validate JSON files using the provided validation script:

```bash
# Validate gap analysis report
python -m models.schemas validate gap_analysis_report.json

# Validate implementation roadmap
python -m models.schemas validate implementation_roadmap.json
```

### External Validation

You can also validate using standard JSON Schema validators:

```bash
# Using ajv-cli (Node.js)
npm install -g ajv-cli
ajv validate -s gap_analysis_report_schema.json -d gap_analysis_report.json

# Using jsonschema (Python)
pip install jsonschema
jsonschema -i gap_analysis_report.json gap_analysis_report_schema.json
```

## Usage Examples

### Reading and Validating Gap Analysis Report

```python
import json
from models.schemas import validate_gap_analysis_report, get_schema_errors

# Read report
with open('outputs/2024-03-15_14-30-00/gap_analysis_report.json', 'r') as f:
    report = json.load(f)

# Validate
try:
    validate_gap_analysis_report(report)
    print("Report is valid!")
    
    # Access data
    print(f"Total gaps: {len(report['gaps'])}")
    print(f"Model used: {report['model_name']}")
    
    # Filter critical gaps
    critical_gaps = [g for g in report['gaps'] if g['severity'] == 'critical']
    print(f"Critical gaps: {len(critical_gaps)}")
    
except Exception as e:
    errors = get_schema_errors(report, GAP_ANALYSIS_REPORT_SCHEMA)
    print(f"Validation errors: {errors}")
```

### Reading and Validating Implementation Roadmap

```python
import json
from models.schemas import validate_implementation_roadmap

# Read roadmap
with open('outputs/2024-03-15_14-30-00/implementation_roadmap.json', 'r') as f:
    roadmap = json.load(f)

# Validate
validate_implementation_roadmap(roadmap)

# Access data
print(f"Immediate actions: {len(roadmap['immediate_actions'])}")
print(f"Near-term actions: {len(roadmap['near_term_actions'])}")
print(f"Medium-term actions: {len(roadmap['medium_term_actions'])}")

# Get all high-effort actions
high_effort = []
for timeframe in ['immediate_actions', 'near_term_actions', 'medium_term_actions']:
    high_effort.extend([a for a in roadmap[timeframe] if a['effort'] == 'high'])

print(f"High-effort actions: {len(high_effort)}")
```

### Generating Reports from JSON

```python
import json
from datetime import datetime

# Read gap analysis report
with open('gap_analysis_report.json', 'r') as f:
    report = json.load(f)

# Generate summary report
print("=" * 60)
print("GAP ANALYSIS SUMMARY")
print("=" * 60)
print(f"Analysis Date: {report['analysis_date']}")
print(f"Policy: {report['input_file']}")
print(f"Model: {report['model_name']}")
print()

# Count by severity
severity_counts = {}
for gap in report['gaps']:
    severity = gap['severity']
    severity_counts[severity] = severity_counts.get(severity, 0) + 1

print("Gaps by Severity:")
for severity in ['critical', 'high', 'medium', 'low']:
    count = severity_counts.get(severity, 0)
    print(f"  {severity.capitalize()}: {count}")

print()
print(f"Covered Subcategories: {len(report['covered_subcategories'])}")
print()

# List critical gaps
critical_gaps = [g for g in report['gaps'] if g['severity'] == 'critical']
if critical_gaps:
    print("CRITICAL GAPS:")
    for gap in critical_gaps:
        print(f"  - {gap['subcategory_id']}: {gap['description'][:60]}...")
```

### Schema Evolution and Versioning

The schemas include a `$schema` field for versioning. When schemas evolve:

1. **Backward-compatible changes** (adding optional fields):
   - Increment minor version
   - Old data remains valid

2. **Breaking changes** (removing fields, changing types):
   - Increment major version
   - Provide migration scripts

Example migration script:

```python
def migrate_v1_to_v2(old_report):
    """Migrate gap analysis report from v1 to v2 schema."""
    new_report = old_report.copy()
    
    # Add new required field with default value
    new_report['input_file_hash'] = "unknown"
    
    # Rename field
    if 'model' in new_report:
        new_report['model_name'] = new_report.pop('model')
    
    return new_report
```
